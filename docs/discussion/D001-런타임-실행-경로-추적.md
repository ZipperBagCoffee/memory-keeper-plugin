# D001 - 런타임 실행 검증 - 검증의 재정의

## Intent (의도)
워크플로우의 "검증" 정의가 불완전하다. 현재 검증은 비평, 의도 부합, 코드 존재 여부까지만 확인하고, "이 코드가 런타임에서 실제로 의도한 대로 실행될 것인가"를 검증하지 않는다. 검증의 정의 자체를 재정의해야 한다.

## Context (배경)
- 현재 워크플로우의 검증 4요소 중 3개만 존재:
  1. 비평 (critique) ✅
  2. 의도 부합 (intent alignment) ✅
  3. 구현 확인 (code exists) ✅
  4. 런타임 실행 검증 (runtime verification) ❌
- 실제 사례: Unity 프로젝트에서 코드를 작성하고 "파일에 있음"을 확인했지만, Boot씬 → Hub씬 전환 시 FindReferences()가 다시 호출되지 않아 기능이 죽어있었음
- 현재 RULES에 "File contains X is NEVER valid verification"이라는 규칙이 있지만, 이를 대체할 구체적 절차가 없음

**문제 정의:**

검증(verification)이란:
- 비평: 이게 좋은 코드인가
- 의도 부합: 유저가 요청한 것과 일치하는가
- 구현 확인: 제대로 코딩되었는가
- **런타임 실행 검증: 실제 실행했을 때 의도한 대로 동작할 것인가**

4번째가 빠져있다. 이것 없이는 검증이 아니다.

**현재 워크플로우가 하는 것:**
```
분석 → 계획 → 구현 → "파일에 코드가 있나" 확인 → 완료 선언
```

**있어야 하는 것:**
```
분석 → 계획 → 구현 → 런타임 실행 검증 → 완료 선언
```

**런타임 실행 검증이란:**
구현한 코드가 실제로 실행되는 경로를 처음부터 끝까지 따라가는 것.

예시 (Unity):
```
Boot씬에서 Play →
GameBootstrapper.Awake() →
싱글톤 생성 →
SceneManager.LoadScene("Hub") →
Hub씬 로드 →
GameFlowManager.Start()는 이미 Boot에서 실행됨 →
FindReferences() 다시 안 불림 →
TitleScreenUI 못 찾음 →
타이틀 입력 무반응
```

이 추적을 하면 "코드는 있지만 실행 안 됨"을 구현 단계에서 잡을 수 있다.

**핵심 질문:**
1. 이 런타임 실행 검증을 워크플로우의 어느 단계에 넣을 것인가
2. Review Agent가 할 것인가, 별도 단계인가
3. 어떤 형식으로 추적 결과를 문서화할 것인가
4. LLM이 실행 없이 이걸 할 수 있는 범위의 한계는 어디인가

### [2026-03-15 20:46] 결론 도출

**Q1 — 어느 단계에 넣을 것인가:**
오케스트레이터가 판단. Work Agent 구현 후, Review Agent 검증 시, Orchestrator 최종 판단 시 — 모든 단계에서 런타임 실행 검증이 필요하다. 단, 각 단계의 깊이는 다를 수 있음.

**Q2 — 누가 하나:**
모두가 한다. 계층 구조:
- **Work Agent:** 구현 후 자기가 쓴 코드의 실행 경로를 먼저 증명
- **Review Agent:** Work Agent의 런타임 실행 검증을 검증하고, 놓친 분기/상태를 찾음
- **Orchestrator (가장 중요):** Intent Guardian으로서 "유저의 의도가 런타임에서 실현되는가"를 최종 판단. 유저가 원한 건 "이게 돌아가는 것"이지 "코드가 파일에 있는 것"이 아니므로, 런타임 실행 검증 없이는 Intent Guardian 역할 자체가 불가능.

**Q3 — 문서화 형식:**
추후 논의 (이번 논의 범위 밖)

**Q4 — LLM의 한계:**
할 수 있을 때도 있고 못할 때도 있다. 하지만:
- **할 수 있는데 안 하는 건 위반이다.**
- 런타임 실행 검증은 "가능하면 하는 옵션"이 아니라 검증의 필수 구성요소
- LLM이 100% 정확하지 못할 수 있지만, 시도 자체를 안 하는 것 = 검증을 안 한 것
- dry run / mental execution으로 정적 추적은 충분히 가능한 경우가 많음

**핵심 결론:**
> 검증 = 비평 + 의도부합 + 구현확인 + **런타임 실행 검증**
> 4번째 없이 "검증 완료"를 선언하는 것은 거짓이다.
> Orchestrator가 이를 최종적으로 확인하는 게이트키퍼 역할을 해야 한다.
